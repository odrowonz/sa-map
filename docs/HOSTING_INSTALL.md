# Установка приложения «Карта системного аналитика» на хостинг (пошагово)

Документ для **начинающего**: что нажимать и в каком порядке. Предполагается типичный **виртуальный хостинг** с **Plesk**, **PHP**, **MariaDB**, **WordPress** уже работает на домене (как на it-senior.kz). Конкретные кнопки в панели могут слегка отличаться у провайдера — ищите аналоги по смыслу.

---

## 0. Что вы в итоге сделаете

**Целевой адрес приложения:** `https://it-senior.kz/sa-map/` (каталог на сервере: `httpdocs/sa-map/`).

1. Создадите **отдельную базу данных** (или пользователя) под приложение карты — **не смешивая** с таблицами WordPress.
2. Загрузите **файлы** приложения в подкаталог **`/sa-map/`** в корне сайта.
3. Настроите **PHP**, **права на файлы**, при необходимости **SSL** и **перезапись URL** (`.htaccess`).
4. Пропишете **подключение к БД** в конфиге приложения на сервере.
5. Добавите **ссылку в меню WordPress** (раздел [11](#11-связка-с-wordpress-и-пункт-меню)).

Пока **готового дистрибутива** из репозитория может не быть — тогда раздел «Загрузка файлов» выполняется **после** появления папки `build/dist/` (см. `build/README.md` в корне проекта).

---

## 1. Что подготовить до начала

| Что нужно | Зачем |
|-----------|--------|
| Доступ в **Plesk** (логин/пароль от панели хостинга) | Управление сайтом, БД, PHP |
| Доступ **FTP** или **SFTP** (логин, пароль, адрес сервера) | Загрузка файлов в `httpdocs` |
| Зафиксированный URL **`https://it-senior.kz/sa-map/`** | Пути в конфиге и загрузка в `httpdocs/sa-map/` |
| Архив с приложением (ZIP) с вашего ПК | То, что выгрузила сборка в `build/dist/` |
| Блокнот для **новых** паролей БД | Сгенерировать сложный пароль и сохранить в менеджере паролей |

**Не храните** пароли в открытых файлах на диске и не вставляйте их в публичные чаты.

## 2. Резервная копия (обязательно перед правками)

1. Войдите в **Plesk** → ваш **домен** → **Резервное копирование** (или «Backup Manager»).
2. Создайте **резервную копию** сайта и базы WordPress.
3. Дождитесь завершения.

Если что-то пойдёт не так, можно откатиться.

---

## 3. Создание базы данных под приложение карты

**Зачем:** таблицы `sa_*` из ТЗ (см. `docs/TZ_...md`, п. 6) должны жить **отдельно** от `wp_*`.

1. Plesk → **Базы данных** (Databases) → **Добавить базу данных**.
2. Укажите **имя БД** (например `user_sa_map` — префикс часто навязывает хостинг).
3. Создайте **пользователя БД** с **сильным паролем**; привяжите пользователя к этой БД с правами **все операции** на эту БД.
4. Запишите четыре значения:
   - **хост** (часто `localhost` или `localhost:3306`);
   - **имя БД**;
   - **пользователь**;
   - **пароль**.

Эти данные позже внесёте в конфиг приложения (например `.env` на сервере — **не** загружайте `.env` с паролем обратно в Git).

### 3.1. Нужно ли что-то «загружать» в базы через phpMyAdmin?

| База | Нужно ли вручную импортировать SQL? |
|------|-------------------------------------|
| **Первая (WordPress)** | **Нет**, если сайт уже работает: таблицы `wp_*` создал сам WordPress. phpMyAdmin — для просмотра, экспорта бэкапа, редко — для восстановления или миграции сайта. Произвольные скрипты в эту БД не заливайте без необходимости. |
| **Вторая (`itsenior_sa_map` и т.п.)** | **Пока приложение не развёрнуто** — можно **ничего не импортировать**; база пустая — нормально. Таблицы `sa_*` появятся после установки **кода** приложения в `httpdocs/sa-map/` (миграции или установочный SQL из репозитория `src/`). **Админка WordPress вторую БД не настраивает** — у WP своя БД, связи нет. |

**Итого:** не «по одному скрипту на каждую БД». WordPress-БД для карты **не трогаете**; во вторую — **один** набор миграций **тогда**, когда выкладываете само приложение (или дождитесь инструкции разработчика). Импортировать что-то во вторую БД сейчас нужно только если вы сознательно кладёте туда готовый дамп от разработчика.

## 4. Куда класть файлы на сервере

У виртуального хостинга корень сайта обычно **`httpdocs`** (или `public_html`).

**Решение проекта:** приложение только в **`httpdocs/sa-map/`** → публичный URL **`https://it-senior.kz/sa-map/`**. Поддомен для этого релиза не используется.

---

## 5. Загрузка файлов по FTP/SFTP

1. Установите клиент: **FileZilla**, **WinSCP** или встроенный FTP в Plesk (**Файлы**).
2. Подключитесь: **хост** (IP или `ftp.ваш-домен.kz`), **пользователь FTP**, **пароль**, порт **21** (FTP) или **22** (SFTP — предпочтительнее).
3. На сервере откройте `httpdocs`.
4. Создайте папку **`sa-map`** (латиница, без пробелов).
5. Загрузите **содержимое** вашего дистрибутива (всё из `build/dist/`) **внутрь** `sa-map/`, так чтобы `index.php` или `index.html` корня приложения лежал по пути `httpdocs/sa-map/...`.

**Права (Linux):** каталоги часто **755**, файлы **644**; папка для загрузок пользователями — по инструкции фреймворка (иногда **775** и отдельный владелец). Если что-то не пишется — уточните у поддержки хостинга.

---

## 6. PHP и расширения

1. Plesk → **домен** → **Настройки PHP** (PHP Settings).
2. Версия **PHP 8.2+** (как в ваших заметках по хостингу), **FastCGI** включён.
3. Увеличьте при необходимости: `upload_max_filesize`, `post_max_size`, `max_execution_time` (для больших ZIP — по рекомендации разработчика приложения).
4. Включите расширения: **pdo_mysql**, **json**, **mbstring**, **openssl** (точный список — из README приложения после появления кода).

---

## 7. SSL (HTTPS)

1. Plesk → **SSL/TLS сертификаты** → **Let's Encrypt** → получить/обновить для домена и поддомена.
2. Включите **принудительный HTTPS** (перенаправление с HTTP), если предлагает панель.

---

## 8. Перезапись URL (если приложение на PHP с роутингом)

Если в корне `sa-map/` лежит фронт + `index.php` роутер:

1. В каталог `sa-map/` поместите **`.htaccess`** из дистрибутива (разработчик должен приложить шаблон).
2. В Plesk должна быть разрешена директива **AllowOverride** для этого каталога (на shared обычно да).

Для **чистого SPA** (только статика) иногда достаточно настроить fallback на `index.html` — зависит от сборки.

---

## 9. Первый запуск и конфигурация

1. На сервере в `sa-map/` создайте файл **`.env`** (или скопируйте из `.env.example` и переименуйте), **не** коммитьте его в Git.
2. Пропишите:
   - `DB_HOST`, `DB_DATABASE`, `DB_USERNAME`, `DB_PASSWORD`;
   - `APP_URL=https://it-senior.kz/sa-map` (или со слэшем в конце — как договоритесь в коде).
3. Если приложение использует миграции: один раз выполните команду миграций **по инструкции разработчика** (часто через веб-установщик или `php artisan migrate` — на shared без SSH может быть веб-скрипт).

---

## 10. Проверка

1. Откройте в браузере: `https://it-senior.kz/sa-map/`.
2. Должна открыться страница входа/приложение без **500** ошибки.
3. Если **500** — Plesk → **Журналы** (Logs) → error log домена; исправьте права, `.env`, версию PHP.

---

## 11. Связка с WordPress и пункт меню

- **Не** устанавливайте файлы приложения **поверх** `wp-admin`, `wp-content`.

### 11.1. Пункт в главном меню (рекомендуется)

1. Войдите в админку WordPress: `https://it-senior.kz/wp-admin/`.
2. Перейдите: **Внешний вид → Меню** (Appearance → Menus).
3. Выберите меню, которое выводится в шапке сайта (часто называется «Primary», «Главное», «Header»).
4. Слева найдите блок **«Произвольные ссылки»** (Custom links).
5. Поле **URL:** `https://it-senior.kz/sa-map/`
6. Поле **Текст ссылки:** например `Карта системного аналитика` или короче — `SA Map`.
7. Нажмите **Добавить в меню**.
8. При необходимости перетащите пункт в нужное место в списке.
9. Нажмите **Сохранить меню**.
10. Откройте главную страницу сайта в режиме инкогнито и проверьте, что новый пункт ведёт на приложение.

**Открытие в новой вкладке:** зависит от темы. Часто в раскрывающихся настройках пункта меню есть галочка «Открывать в новой вкладке»; если нет — можно использовать плагин для расширенных настроек меню или оставить открытие в той же вкладке.

### 11.2. Страница с кнопкой (дополнительно)

Создайте **Страницы → Добавить новую**: краткий текст о сервисе и кнопку/ссылку на `https://it-senior.kz/sa-map/`. Страницу можно добавить в меню как обычную страницу, а не произвольную ссылку — но тогда пользователь сначала попадёт на «лендинг» WP, а не сразу в приложение.

### 11.3. Учётные записи

Отдельная **единая регистрация** WP + приложение — отдельная задача (плагин, OAuth). В базовой схеме пользователи приложения **свои**, меню только **направляет** на URL приложения.

---

## 12. Обновление приложения

1. Снова **бэкап** БД и файлов `sa-map/`.
2. Загрузите новые файлы **поверх** старых (кроме `.env` и `uploads/` — не затирайте пользовательские файлы).
3. Запустите миграции БД, если есть.

---

## 13. Если нет SSH

На части тарифов SSH **запрещён**. Тогда:

- миграции и кэш — через **веб-интерфейс** или локальный запуск с последующей выгрузкой **готовой** БД (редко);
- зависимости PHP — заливать уже собранный **`vendor/`** с локальной машины.

Это нормально для shared-хостинга; главное — **не хранить секреты** внутри загружаемого архива публично.

---

## 14. Куда обращаться за помощью

- **Поддержка хостинга** — если не подключается FTP, не создаётся БД, ошибки SSL.
- **Документация фреймворка** приложения — после появления исходного кода.
